# Overview
Prismatica Relay is a high‑performance, parallelized metadata aggregation service built with **Sanic** and **Pydantic**.  
It receives a client query, splits it into independent sub‑requests, fetches data from multiple upstream sources concurrently, and returns a deduplicated, ranked result set with ultra‑low latency.

The service is designed for modern web APIs that require:
- Sub‑millisecond response times
- Horizontal scalability across shards
- Graceful handling of partial failures
- Strong type safety and automatic OpenAPI documentation

# Features
- **Request splitting** – Transform a single client payload into N granular `SubRequest` objects.
- **Async fetching** – Use `httpx.AsyncClient` to fire concurrent HTTP calls with per‑request timeouts.
- **Result aggregation** – De‑duplicate, merge, and rank metadata items from heterogeneous sources.
- **Robust error handling** – Distinguish retryable upstream errors, return partial results with a `partial_success` flag, and provide detailed JSON error payloads.
- **Pydantic validation** – Enforce schemas for inbound queries and outbound responses.
- **Sanic server** – Lightweight async framework with built‑in health‑check endpoints.
- **Dependency injection** – Core services are injected into routes for testability.
- **Comprehensive test suite** – Unit tests cover splitter, aggregator, and fetcher logic.
- **CI pipeline** – Automated linting and testing on every push and pull request.

# Quickstart
The following example demonstrates a minimal client that posts a query to the `/metadata` endpoint and prints the aggregated result.

```python
import asyncio
import json
from httpx import AsyncClient

async def main() -> None:
    async with AsyncClient(base_url="http://127.0.0.1:8000") as client:
        payload = {
            "query": "latest",
            "filters": {"tags": ["science", "technology"]},
            "limit": 10
        }
        response = await client.post("/metadata", json=payload)
        response.raise_for_status()
        data = response.json()
        print(json.dumps(data, indent=2))

if __name__ == "__main__":
    asyncio.run(main())
```

Run the server in one terminal:

```bash
prismatica-relay
```

# Architecture
```
┌──────────────────┐
│  RequestSplitter   │
└──────────────────┘
         │          
         ▼          
┌──────────────────┐
│  ResultAggregator  │
└──────────────────┘
         │          
         ▼          
┌──────────────────┐
│      SanicApp      │
└──────────────────┘
```

The data flow proceeds as follows:
1. **Client** → POST `/metadata` with JSON query.  
2. **Sanic route** validates payload and invokes `RequestSplitter.split`.  
3. **RequestSplitter** emits a list of `SubRequest` objects.  
4. **AsyncFetcher** performs N concurrent HTTP calls using `httpx.AsyncClient`.  
5. Each response is wrapped in an `ApiResponse`.  
6. **ResultAggregator** receives all `ApiResponse` objects, deduplicates, merges, and ranks them.  
7. The final `AggregatedResult` is serialized and returned to the client.

# API Reference

## Core Models (`src/core/models.py`)

### `MetadataItem`
```python
class MetadataItem(BaseModel):
    id: str
    title: str
    tags: List[str]
    thumbnail_url: HttpUrl
    source: str
```
Represents a single metadata entry returned to the client.

### `SubRequest`
```python
class SubRequest(BaseModel):
    endpoint: HttpUrl
    params: Dict[str, Any]
    timeout_ms: int = 2000
```
Describes an individual upstream request generated by the splitter.

### `ErrorDetail`
```python
class ErrorDetail(BaseModel):
    code: str
    message: str
    retryable: bool
```
Encapsulates error information for non‑2xx upstream responses.

### `ApiResponse`
```python
class ApiResponse(BaseModel):
    status: int
    payload: List[MetadataItem] = []
    error: Optional[ErrorDetail] = None
```
Standardizes the shape of data received from each upstream source.

## RequestSplitter (`src/core/splitter.py`)

### `RequestSplitter.split(query: dict) -> List[SubRequest]`
Divides a high‑level client query into independent `SubRequest` objects after validating the input schema.

### `RequestSplitter.validate_subrequest(sub: SubRequest) -> bool`
Ensures that a generated `SubRequest` conforms to the expected upstream schema; returns `True` if valid.

## ResultAggregator (`src/core/aggregator.py`)

### `ResultAggregator.aggregate(responses: List[ApiResponse]) -> Dict[str, Any]`
Merges a collection of `ApiResponse` objects into a single payload, handling deduplication, conflict resolution, and ranking. Returns a dictionary with keys `partial_success` and `results`.

### `ResultAggregator.deduplicate(items: List[MetadataItem]) -> List[MetadataItem]`
Removes duplicate `MetadataItem` entries based on a deterministic fingerprint derived from `id` and `source`.

## SanicApp (`src/api/routes.py`)

### `create_app() -> sanic.Sanic`
Bootstraps the Sanic server, configures logging, CORS, and registers the error‑handling middleware.

### `add_routes(app: sanic.Sanic) -> None`
Registers the `/metadata` POST endpoint and the `/healthz` GET endpoint with the provided Sanic application instance.

## Middleware (`src/middleware/error_handler.py`)

### `error_handler(request, exception) -> sanic.response.HTTPResponse`
Converts validation errors, upstream failures, and unexpected exceptions into structured JSON responses with appropriate HTTP status codes.

# Contributing
We welcome contributions from the community. Follow these steps to submit a change:

1. **Fork** the repository at `github.com/asmit25805/prismatica-relay`.
2. **Create** a new branch for your feature or bug fix.
3. **Write** tests in the `tests/` directory; each test file must contain at least six test functions exercising real behaviour.
4. **Run** the full test suite locally with `pytest -q`.
5. **Commit** your changes with clear, descriptive messages.
6. **Open** a pull request targeting the `main` branch. CI will automatically lint and test your contribution.

Please ensure that new code adheres to the project's coding standards:
- Type annotations on all public functions.
- Pydantic models for any new data structures.
- No stub implementations; every method must contain functional logic.
- Keep the line count of new modules above 100 lines where applicable.

For any questions, refer to the project's issue tracker on GitHub.
